Functionbeat overviewedit

Functionbeat is an Elastic Beat that you deploy as a function in your serverless environment to collect data from cloud services and ship it to the Elastic Stack.

Version 7.8.1 supports deploying Functionbeat as an AWS Lambda service or Google Cloud Function. It responds to triggers defined for the following event sources:

  • Amazon CloudWatch Logs
  • Amazon Simple Queue Service (SQS)
  • Amazon Kinesis
  • Google Cloud Pub/Sub
  • Google Cloud Storage

Functionbeat is an Elastic Beat. It’s based on the libbeat framework. For more information, see the Beats Platform Reference.

The following sections explore some common use cases for Functionbeat:

  • Monitor cloud deployments
  • Perform event-driven processing

Monitor cloud deploymentsedit

You can deploy Functionbeat on your serverless environment to collect logs and metrics generated by cloud services and stream the data to the Elastic Stack for centralized analytics.

Monitor AWS services with CloudWatch logsedit

You can deploy Functionbeat as a Lambda function on AWS to receive events from a Cloudwatch Log group, extract and structure the relevant fields, then stream the events to Elasticsearch.

The processing pipeline for this use case typically looks like this:

  1. Functionbeat runs as a Lambda function on AWS and reads the data stream from a Cloudwatch Log group.
  2. Beats processors, such as dissect and drop_fields, filter and structure the events.
  3. Optional ingest node pipelines in Elasticsearch further enhance the data.
  4. The structured events are indexed into an Elasticsearch cluster.

Perform event-driven processingedit

You can use Functionbeat to implement event-driven processing workflows with cloud messaging queues and the Elastic Stack. Functionbeat responds to event triggers from AWS Kinesis and SQS.

Analyze application data from SQSedit

For applications that send JSON-encoded events to an SQS queue, Functionbeat can listen for, ingest, and decode JSON events prior to shipping them to Elasticsearch, where you can analyze the streaming data.

The processing pipeline for this use case typically looks like this:

  1. Functionbeat runs as a serverless shipper and listens to an SQS queue for application events.
  2. The Beats decode_json_fields processor decodes JSON strings and replaces them with valid JSON objects.
  3. Optional ingest node pipelines in Elasticsearch further enhance the data.
  4. The events are indexed into an Elasticsearch cluster.
